# AAP Deployment Guide: Applying CaC to Ansible Automation Platform

> How to take the YAML generated by `cac-author` and apply it to a live AAP controller.

---

## Overview

The `cac-author` sub-agent outputs `infra.controller_configuration`-compatible YAML under `aap_config/job_templates/`. These files are not applied automatically — you run a deployment playbook that pushes them to the AAP controller API.

The tool that does the pushing is the `infra.controller_configuration` collection, specifically its `dispatch` role, which reads your YAML and creates or updates objects in the controller via its REST API.

---

## Prerequisites

### On your workstation

- Ansible 2.16+ (`ansible --version`)
- The `infra.controller_configuration` collection:

  ```bash
  ansible-galaxy collection install infra.controller_configuration
  ```

- Network access to your AAP controller host on port 443

### In AAP

- An AAP user account with **System Administrator** or **Organization Admin** role — the collection creates and modifies objects; read-only accounts will fail
- The **project** already exists in the controller and is synced (the `cac-author` output references a project by name; it must exist before you apply)
- The **inventory** already exists in the controller (same reason)
- The **machine credential** already exists in the controller

> If any of these don't exist yet, create them manually in the AAP UI first, or add them to your CaC pipeline as separate `controller_credentials` and `controller_inventories` files. The `cac-author` agent is scoped to job templates and surveys only.

## Controller Inventory File

The deployment playbook targets the AAP controller itself, not your managed hosts. You need a small inventory file pointing at the controller:

```ini
# inventory/controller.ini
[automationcontroller]
controller.example.com

[automationcontroller:vars]
ansible_connection=local
```

Using `ansible_connection=local` is the standard pattern — the collection talks to the controller via its REST API, so there is no SSH connection to the controller host.

---

## Authentication

The collection authenticates to the controller API using variables. Set these via environment variables (never hardcode them):

```bash
export CONTROLLER_HOST=https://controller.example.com
export CONTROLLER_USERNAME=admin
export CONTROLLER_PASSWORD=<your password>
export CONTROLLER_VERIFY_SSL=true
```

Or pass them as extra vars at runtime:

```bash
ansible-playbook deploy_cac.yml \
  -e controller_host=https://controller.example.com \
  -e controller_username=admin \
  -e controller_password="${AAP_PASSWORD}" \
  -e controller_validate_certs=true
```

For production use, store the password in Ansible Vault or retrieve it from a secrets manager — do not pass it in plain text on the command line.

## Deployment Playbook

Create this once in your repo at `aap_config/deploy_cac.yml`:

```yaml
---
- name: Apply Controller-as-Code to AAP
  hosts: automationcontroller
  connection: local
  gather_facts: false
  vars_files:
    - "{{ lookup('fileglob', 'job_templates/*.yml') | list }}"
  roles:
    - role: infra.controller_configuration.dispatch
      vars:
        controller_host: "{{ lookup('env', 'CONTROLLER_HOST') }}"
        controller_username: "{{ lookup('env', 'CONTROLLER_USERNAME') }}"
        controller_password: "{{ lookup('env', 'CONTROLLER_PASSWORD') }}"
        controller_validate_certs: "{{ lookup('env', 'CONTROLLER_VERIFY_SSL') | default(true) }}"
```

> The `dispatch` role inspects which `controller_*` variables are defined in the loaded var files and calls the appropriate sub-roles automatically. You do not need to call `controller_job_templates` directly.

## Running a Deployment

### Dry run first (always)

The collection supports check mode. Run this before applying to production:

```bash
ansible-playbook aap_config/deploy_cac.yml \
  -i inventory/controller.ini \
  --check \
  --diff
```

Check mode will show you which objects would be created or updated without touching the controller.

### Apply to a single spec's config

```bash
ansible-playbook aap_config/deploy_cac.yml \
  -i inventory/controller.ini \
  -e "controller_templates_enforce_defaults=false" \
  --limit automationcontroller
```

### Apply all CaC in the repo

```bash
ansible-playbook aap_config/deploy_cac.yml \
  -i inventory/controller.ini
```

This loads every `aap_config/job_templates/*.yml` file and applies them all. The `dispatch` role is idempotent — objects that already match are left unchanged, objects that differ are updated.

## Validating the Deployment

After applying, verify in the AAP UI:

1. **Templates** → confirm the job template appears with the correct name, playbook path, inventory, and credentials
2. **Templates → Survey** → open the survey and confirm every field matches spec §4:
   - Variable names match `defaults/main.yml` exactly
   - Types are correct (integer fields reject strings, password fields are masked)
   - Required/optional flags are correct
   - Defaults are populated where expected

Via CLI:

```bash
aap job template list --format json | jq '.[] | select(.name=="<job template name>")'
```

## Rollback

The `infra.controller_configuration` collection does not have a built-in rollback. To undo a bad deployment:

**Option 1 — Revert the YAML and redeploy**

```bash
git revert HEAD          # reverts the bad aap_config/ commit
ansible-playbook aap_config/deploy_cac.yml -i inventory/controller.ini
```

Because the collection is idempotent, re-applying the previous state restores the controller objects to their prior definition.

**Option 2 — Delete the job template manually**

If the template was newly created (not an update), delete it in the AAP UI under **Templates**, then re-apply the correct version.

> Deleting a job template also deletes its survey — there is no way to delete the survey independently. If you only need to fix survey fields, edit the YAML and redeploy; do not delete the template.

## Per-Environment Deployments

If you manage multiple environments (dev, staging, prod) with separate AAP controllers, use separate inventory files and pass the correct one at deploy time:

```bash
# Deploy to dev controller
ansible-playbook aap_config/deploy_cac.yml \
  -i inventory/controller-dev.ini

# Deploy to prod controller
ansible-playbook aap_config/deploy_cac.yml \
  -i inventory/controller-prod.ini
```

Keep environment-specific variable overrides (different credential names, different EE images) in host_vars files per controller:

```
inventory/
  controller-dev.ini
  controller-prod.ini
  host_vars/
    controller-dev.example.com.yml   # dev-specific overrides
    controller-prod.example.com.yml  # prod-specific overrides
```

## CI/CD Integration

To apply CaC automatically on merge to `main`, add a pipeline stage after the existing lint/test stages:

```yaml
# .github/workflows/sdd-validate.yml (add this job)
deploy-cac:
  needs: [lint, molecule]
  runs-on: self-hosted
  environment: aap-production
  steps:
    - uses: actions/checkout@v4
    - name: Install collection
      run: ansible-galaxy collection install infra.controller_configuration
    - name: Apply CaC
      env:
        CONTROLLER_HOST: ${{ secrets.CONTROLLER_HOST }}
        CONTROLLER_USERNAME: ${{ secrets.CONTROLLER_USERNAME }}
        CONTROLLER_PASSWORD: ${{ secrets.CONTROLLER_PASSWORD }}
      run: |
        ansible-playbook aap_config/deploy_cac.yml \
          -i inventory/controller.ini
```

Gate the `deploy-cac` job on a successful `molecule` run so broken playbooks are never wired up in the controller.

## Relationship to the SDD Workflow

```
spec (approved) ──► cac-author ──► aap_config/job_templates/<SPEC-ID>.yml
                                          │
                                          ▼
                              ansible-playbook deploy_cac.yml
                                          │
                                          ▼
                              AAP Controller (job template + survey live)
```

The generated YAML is the single source of truth for what the controller should look like. If someone edits the job template directly in the AAP UI, the next `deploy_cac.yml` run will overwrite their changes — this is intentional. All changes go through the spec → CaC → deploy pipeline.
